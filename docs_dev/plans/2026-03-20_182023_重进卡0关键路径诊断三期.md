生成时间：2026-03-20 18:20:23

## 任务背景
- 现象：世界重进卡在“Preparing spawn area: 0%”，已缩小到“仅建立 1:1 连接且未激活”也可复现。
- 已有隔离：区块生命周期回放已加启动/停服门控，但问题仍存在，说明可能在更靠前的启动关键路径。
- 本轮目标：补齐启动期“可观测性”，定位阻塞在 `SavedData load` 还是 `SERVER_STARTED 后 tick 推进`。

## 方案详情

### 现状分析
- `LinkSavedData`、`CrossChunkDispatchQueueSavedData`、`PendingRetireQueueSavedData` 的 `load(...)` 当前无耗时与规模日志。
- `LinkNodeLifecycleDispatchEvents` 已有诊断状态与生命周期日志，但缺少：
  - `SERVER_STARTING -> SERVER_STARTED` 时长；
  - `SERVER_STARTED` 后首段 tick 心跳。

### 技术方案
- 仅新增诊断日志，不改业务流程、不改判定语义：
  1. 三个 `SavedData.load(...)` 记录读取耗时与关键规模字段；
  2. 生命周期桥接器新增启动阶段时间戳、started 时长日志；
  3. 新增 `START_SERVER_TICK` 监听，在首 200 tick 每 20 tick 打点一次，确认主线程是否持续推进。
- 日志统一使用 `[Diag]` 前缀，便于从 `latest.log` 过滤。

### 影响范围
- `src/main/java/com/makomi/data/LinkSavedData.java`
- `src/main/java/com/makomi/data/CrossChunkDispatchQueueSavedData.java`
- `src/main/java/com/makomi/data/PendingRetireQueueSavedData.java`
- `src/main/java/com/makomi/data/LinkNodeLifecycleDispatchEvents.java`

### 语义对齐说明
- 命名：不新增业务术语字段，保持现有 `triggerSource/core` 映射语义不变。
- 方向：不涉及来源/目标写入逻辑，方向校验链路不变。
- 文案：新增内容仅为诊断日志，不暴露旧术语到用户命令输出。
- 兼容：仅增加只读观测，不改存档结构与持久化键名。
- 残余风险：日志量略增，已通过限频控制（启动窗口固定、每 20 tick 打点）。

## 原子步骤清单

### 步骤 1：补齐三类 SavedData 的 load 诊断
- **操作对象**：三个 `SavedData` 类的 `load(...)`
- **具体动作**：记录 `startNs`，在返回前输出耗时毫秒与核心计数。
- **预期结果**：可判定是否卡在数据反序列化/索引恢复阶段。
- **关键里程碑**：是

### 步骤 2：补齐服务端启动阶段诊断
- **操作对象**：`LinkNodeLifecycleDispatchEvents`
- **具体动作**：新增 `SERVER_STARTING` 标记开始时间，`SERVER_STARTED` 输出耗时；注册 `START_SERVER_TICK` 心跳日志。
- **预期结果**：可判定是否进入 started、是否持续推进 tick。
- **关键里程碑**：是

### 步骤 3：本地回归测试
- **操作对象**：Gradle 测试任务
- **具体动作**：执行 `.\gradlew.bat test testIntegration testClient testSlow testApiLegacy --no-daemon`。
- **预期结果**：确认诊断改动未引入回归。
- **关键里程碑**：是

## 预期结果
- 新增日志可将“卡 0%”定位到以下三类之一：
  1. 卡在某个 `SavedData.load(...)`；
  2. `SERVER_STARTING -> SERVER_STARTED` 异常耗时；
  3. 已 started 但 tick 未推进/推进后卡在后续路径。
