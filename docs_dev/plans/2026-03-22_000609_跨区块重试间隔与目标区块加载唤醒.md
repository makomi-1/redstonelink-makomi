# 跨区块重试间隔与目标区块加载唤醒

生成时间：2026-03-22 00:06:09
文件名：2026-03-22_000609_跨区块重试间隔与目标区块加载唤醒.md

## 任务背景

当前跨区块持久 pending 已支持长期保留与退避重试，但首次失败后仍会按每 tick 热循环尝试，且目标区块后续加载时不会主动撤销等待窗口。用户希望新增可配置的初始重试间隔，并在目标区块加载时主动唤醒对应 pending，使长期保留与及时送达之间达到更稳定的平衡。

## 方案详情

### 现状分析

- `CrossChunkDispatchService` 当前仅在不限时 pending 达到退避阈值后，通过 `lastAttemptTick + persistentBackoffTicks` 控制后续等待。
- 首次失败到进入退避阈值前没有独立的等待窗口，因此持久 pending 在目标区块尚未加载时会频繁重试。
- `LinkNodeLifecycleDispatchEvents` 已统一接管 `CHUNK_LOAD` 生命周期事件，适合复用为 pending 唤醒入口。

### 技术方案

- 在跨区块配置中新增 `crosschunk.retry.initialRetryIntervalTicks`，默认值设为 `10` tick。
- 为 `RetryState` 增加 `nextEligibleTick`，统一表达“下一次允许派发的最早 tick”：
  - 未进入持久退避前，失败后按 `initialRetryIntervalTicks` 推进。
  - 达到持久退避阈值后，失败后按 `persistentBackoffTicks` 推进。
- 新增 `notifyTargetChunkLoaded(...)`：
  - 按目标维度与目标区块匹配当前 pending；
  - 仅清除等待态，不重置 `attempts`；
  - 保留原有告警、错误、退避累计语义。
- 在 `CHUNK_LOAD` 路径接入该唤醒逻辑，使目标区块一旦可用，等待中的 pending 能在下一轮调度立即重试。

## 语义对齐说明

- 命名统一保持 `triggerSource/core`，不新增旧术语。
- 仅影响来源到目标的派发重试节奏，不改变方向合法性与 `LinkNodeType` 映射。
- 新增配置属于增量扩展，默认值生效时无需兼容读取旧键。
- 残余风险：若手工配置过大重试间隔，会让首次送达明显变慢；通过 `CHUNK_LOAD` 唤醒降低该风险。

## 预期结果

跨区块持久 pending 在默认配置下不再每 tick 热循环，而是先按较短初始间隔重试；若目标区块期间加载，则等待会被主动解除，从而兼顾性能与到达时效。
