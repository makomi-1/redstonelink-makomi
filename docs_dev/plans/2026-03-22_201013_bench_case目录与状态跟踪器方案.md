# bench case 目录与状态跟踪器方案

生成时间：2026-03-22 20:10:13
文件名：2026-03-22_201013_bench_case目录与状态跟踪器方案.md

## 任务背景
当前 bench suite 会把每个 case 世界直接放到 external dedicated server 根目录，目录结构较乱。同时，现有运行态只支持读取节点“当前一次性快照”，缺少可复用的历史状态采样层，不利于后续做命令诊断与物品状态面板。

## 方案详情

### 现状分析
- `run-bench-suite.ps1` 当前把模板世界直接复制到 `ServerRoot\<worldName>`，并把 `level-name` 直接写成 `<worldName>`。
- 命令侧当前仅能通过 `node get` 读取在线 `core` 的即时运行态：`configuredMode/effectiveMode/active/resolvedStrength/output/maxSources`。
- 裁决链路已有 `TimeKey(tick, slot)` 结构，默认粒度是 tick，slot 预留未来细分，可直接作为状态采样的时间轴基础。
- `triggerSource` 当前缺少统一的“输入功率/输出功率”读取抽象；只有局部节点（如 sync emitter）持有最近观测强度等状态。

### 技术方案
- 本轮实现：
  1. bench suite 的 case 世界物理目录统一切到 `ServerRoot\rl-cases\<worldName>`。
  2. 写入 `server.properties` 时，`level-name` 改为相对路径 `rl-cases/<worldName>`。
  3. 使用说明同步更新为 `rl-cases` 目录结构。
- 状态跟踪器设计：
  1. 分成两层：`NodeRuntimeProbe`（即时快照）与 `NodeStateTraceSampler`（历史采样）。
  2. `Probe` 负责统一读取 `triggerSource/core` 的当前状态，不做历史缓存。
  3. `Sampler` 只对少量节点动态挂载，按“每 N 个时间粒度”采样并写入 ring buffer。
  4. 命令读取与未来物品状态面板都统一消费 `Probe/Sampler` 的 DTO，不各自重复拼状态。
- 动态挂载结论：
  1. 历史采样器应做成“动态挂载到某个节点”，而不是全局常驻。
  2. 原因是 bench/诊断通常只观察少量热点节点；全量常驻会带来无意义的 tick 成本与内存开销。
  3. 即时状态不需要挂载，直接走 `Probe` 即可；只有历史窗口才需要 `Sampler` 挂载。

### 状态跟踪器建议接口
- 命令建议根：`/redstonelink node trace ...`
- 第一阶段建议支持：
  1. `mount <type> <serial> [every] [capacity]`
  2. `latest <type> <serial>`
  3. `read <type> <serial> [limit]`
  4. `unmount <type> <serial>`
  5. `list`
- DTO 字段建议：
  1. 通用：`type/serial/online/tick/slot/active/inputPower/outputPower`
  2. `core` 扩展：`configuredMode/effectiveMode/resolvedStrength/maxSources`
  3. `triggerSource` 扩展：`lastObservedInputPower/lastDispatchedPower`
- 时间粒度建议：
  1. 第一阶段直接按 `tick` 采样。
  2. 若未来确实需要更细粒度，再复用现有 `TimeKey.slot` 扩展。

### 语义对齐说明
- 本轮代码改动只涉及 bench 运维脚本路径，不改变 `triggerSource/core` 业务语义。
- 状态跟踪器方案中的节点类型统一使用 `triggerSource/core`，与 `LinkNodeType.TRIGGER_SOURCE/CORE` 一一对应。
- 输入输出方向不变：`triggerSource` 为来源，`core` 为目标。
- 未引入旧术语回写，也未新增 `com.makomi.api.v1.*` 依赖。
- 残余风险：`level-name=rl-cases/<worldName>` 需以实际 dedicated server 做一次 smoke test 确认。
