# 常驻限制下线对象与 linker 序号

生成时间：2026-03-15 11:49:22
文件名：2026-03-15_114922_常驻限制下线对象与linker序号.md

## 任务背景
当前 `crosschunk whitelist` 的 resident 语义允许“已分配且未退役”序号进入常驻集合。  
这会把下线节点、以及仅存在于 linker 物品但未上线为方块节点的序号写入 resident，导致“命令成功但常驻不可生效”的歧义。

## 方案详情

### 现状分析
- 常驻入口：`whitelist add ... resident` 与 `whitelist set ... resident`。
- 当前校验：仅校验序号 active（allocated 且 not retired）。
- 缺口：resident 未要求节点在线（`findNode` 命中）。

### 技术方案
- 在 resident 分支新增“在线限制”：
  - `add ... resident`：单条序号必须在线。
  - `set ... resident`：批量序号全部必须在线，存在离线即拒绝并返回列表。
- 非 resident 分支保持现状，不改变白名单基础能力。

### 影响范围
- 命令实现：`src/main/java/com/makomi/command/crosschunk/CrossChunkCommandRegistry.java`
- 语言键：`src/main/resources/assets/redstonelink/lang/zh_cn.json`、`src/main/resources/assets/redstonelink/lang/en_us.json`
- 使用说明：`docs_dev/使用说明.md`

## 原子步骤清单

### 步骤 1：resident 单条限制
- **操作对象**：`CrossChunkCommandRegistry`
- **具体动作**：为 `executeWhitelistAddInternal(..., resident=true)` 增加在线校验
- **预期结果**：下线对象与 linker 离线序号不可加入 resident
- **关键里程碑**：是

### 步骤 2：resident 批量限制
- **操作对象**：`CrossChunkCommandRegistry`
- **具体动作**：为 `executeWhitelistSet(... resident)` 增加离线列表校验
- **预期结果**：批量 resident 命令在存在离线序号时拒绝执行
- **关键里程碑**：是

### 步骤 3：反馈与说明同步
- **操作对象**：语言文件与使用说明
- **具体动作**：新增失败提示键并补充使用约束
- **预期结果**：用户可明确知道 resident 失败原因
- **关键里程碑**：否

### 步骤 4：回归验证
- **操作对象**：Gradle 测试任务
- **具体动作**：执行测试验证无回归
- **预期结果**：构建与测试通过
- **关键里程碑**：是

## 预期结果
- resident 行为收紧为“仅在线节点可设置”。
- 下线对象与 linker 离线序号不再产生“假成功”的 resident 记录。
- 白名单与常驻语义边界更清晰，后续排障成本降低。
