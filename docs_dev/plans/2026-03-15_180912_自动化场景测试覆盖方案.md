# 自动化场景测试覆盖方案

生成时间：2026-03-15 18:09:12
文件名：2026-03-15_180912_自动化场景测试覆盖方案.md

## 任务背景
当前项目在配置解析、跨区块调度、命令系统、GUI 出入口等路径已具备基础测试，但自动化场景覆盖仍存在结构性缺口：
- 配置组合与入口条件缺少系统化交叉验证。
- 跨区块关键处理逻辑已有单测，但策略联动与边界场景覆盖不足。
- 命令系统出入口缺少“可重构友好”的契约测试层。
- GUI 相关逻辑在 client 标签下缺少有效自动化用例。

本方案目标是在不引入额外 API 依赖、不改变现有业务语义的前提下，补齐自动化测试覆盖闭环，并保持后续架构优化空间。

## 方案详情

### 现状分析
1. 已有测试标签分层（stable-core / integration / client / slow）与执行任务，但 client 套件基本空缺，扩展套件覆盖不均衡。
2. 配置入口分散在 RedstoneLinkConfig 的基础配置、交互门禁与跨区块配置解析路径。
3. 命令入口集中在 ModCommands 与 CrossChunkCommandRegistry，包含权限门禁、confirm 后缀与语义 type 解析。
4. GUI 出入口分布于 PairingNetwork / RedstoneLinkClient / Screen 层，已有网络 payload 契约测试，但缺少 GUI 输入与命令拼装测试。

### 技术方案
1. 测试分层策略
- stable-core：覆盖纯逻辑、配置解析、命令输入契约、关键守卫分支。
- integration：覆盖跨组件联动路径（配置+数据+命令/调度）。
- client：覆盖 GUI 输入解析与命令拼装等客户端纯逻辑。
- slow：保留高压与长时场景，避免影响默认门禁。

2. 配置组合与入口交叉验证
- 补 RedstoneLinkConfig 解析矩阵测试：基础命令权限、crosschunk 命令权限、交互门禁布尔组合。
- 补 canOpenPairingByHeldItem/canOpenPairingByLinker/canOpenPairingByPlacedBlock 的场景表驱动测试。

3. 跨区块关键处理逻辑
- 基于 CrossChunkDispatchService 现有用例补“relay/forceLoad 策略切换、TTL 过期、按来源限流、resident 票据同步”的组合测试。
- 与 LinkedTargetDispatchService 补联动断言，验证跨区块接管统计与提示数据一致性。

4. 命令系统出入口（保留架构优化空间）
- 以“输入契约与语义结果”为核心，优先覆盖：confirm 后缀解析、type 语义映射、权限门禁关键分支。
- 避免把测试强绑定到命令树内部节点结构，减少后续命令树重构时的测试脆弱性。

5. GUI 出入口与关键逻辑
- 新增 client 标签测试，覆盖 AbstractMultiPairingScreen 的输入解析与限制逻辑。
- 覆盖 TriggerSourcePairingScreen/CorePairingScreen 的 set_links 命令拼装规则（空输入、多目标自动 confirm、清空命令）。
- 保持网络映射路径由既有 PairingNetwork payload/dispatch 测试兜底。

6. 扩展与回归
- 补充测试用例模板（正常/边界/异常/回归）并按标签归类。
- 执行 ./gradlew test、./gradlew testIntegration、./gradlew testClient、./gradlew testSlow、./gradlew testCoverage 回归验证。

### 影响范围
- 测试代码：src/test/java/com/makomi/config、src/test/java/com/makomi/data、src/test/java/com/makomi/command（新增）、src/test/java/com/makomi/client（新增）。
- 构建与验证：沿用现有 Gradle 测试任务，不新增业务运行依赖。
- 业务代码：原则上不改；若测试可测性受阻，再单独提出最小化可测性改造方案。

## 原子步骤清单

### 步骤 1：建立覆盖矩阵
- **操作对象**：现有测试目录与标签任务
- **具体动作**：盘点现有用例并形成“配置/跨区块/命令/GUI”覆盖矩阵及缺口清单
- **预期结果**：得到可执行的补测清单与优先级
- **关键里程碑**：是

### 步骤 2：补配置组合与入口测试
- **操作对象**：config 测试包
- **具体动作**：新增参数化/表驱动用例，覆盖解析边界与开屏门禁组合
- **预期结果**：配置与入口条件具备稳定回归保护
- **关键里程碑**：是

### 步骤 3：补跨区块关键逻辑测试
- **操作对象**：CrossChunkDispatchServiceTest、LinkedTargetDispatchServiceTest
- **具体动作**：补策略矩阵、过期处理、限流与 resident 同步相关场景
- **预期结果**：跨区块关键路径覆盖更完整
- **关键里程碑**：是

### 步骤 4：补命令系统契约测试
- **操作对象**：命令测试包（新增）
- **具体动作**：围绕 confirm/type/权限门禁补契约测试，不绑定命令树内部实现细节
- **预期结果**：命令出入口回归稳定，保留重构空间
- **关键里程碑**：是

### 步骤 5：补 GUI 逻辑测试
- **操作对象**：client 测试包（新增）
- **具体动作**：覆盖 GUI 输入解析、命令拼装与边界行为
- **预期结果**：GUI 出入口关键逻辑进入自动化回归
- **关键里程碑**：是

### 步骤 6：执行验证与归档
- **操作对象**：Gradle 测试任务与覆盖率报告
- **具体动作**：按标签分层执行测试并输出通过/未覆盖项
- **预期结果**：形成可追溯的自动化测试闭环
- **关键里程碑**：是

## 预期结果
1. 自动化测试覆盖从“点状”提升为“分层场景化”覆盖。
2. 配置、跨区块、命令、GUI 四类高频变更路径具备回归保护。
3. 命令测试策略不锁死架构实现细节，为后续命令树优化留出空间。
4. 在不引入 com.makomi.api.v1.* 新依赖前提下，完成当前阶段测试体系扩展。
