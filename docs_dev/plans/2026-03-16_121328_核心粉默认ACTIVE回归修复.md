# 核心粉默认ACTIVE回归修复

生成时间：2026-03-16 12:13:28
文件名：2026-03-16_121328_核心粉默认ACTIVE回归修复.md

## 任务背景
在顶面核心红石粉场景中出现“未接收任何 mod 激活信号，但 F3 的 active 长期保持 true，且与位置相关”的现象。用户已确认采用最小修复路径：恢复核心粉方块默认态中的 `ACTIVE=false`，并补充回归测试防止再次回归。

## 方案详情

### 现状分析
- `LinkRedstoneDustCoreBlock` 构造器当前仅设置 `SUPPORT_FACE`，未显式设置 `ACTIVE` 默认值。
- 顶面逻辑会频繁经过原版 wire 计算路径，若默认态不稳定，可能出现位置相关的观测态污染。
- 当前测试仅覆盖属性名契约，未覆盖默认 `ACTIVE` 值契约。

### 技术方案
- 在 `LinkRedstoneDustCoreBlock` 构造器中显式恢复 `ACTIVE=false` 默认值，确保所有由默认态派生的状态都以未激活为起点。
- 在 `LinkRedstoneDustCoreStateContractTest` 增加“默认 ACTIVE 必须为 false”的契约测试。
- 执行全标签自动化测试作为回归门禁，确保无功能回归。

### 影响范围
- `src/main/java/com/makomi/block/LinkRedstoneDustCoreBlock.java`
- `src/test/java/com/makomi/block/LinkRedstoneDustCoreStateContractTest.java`

## 原子步骤清单

### 步骤 1：恢复默认态约束
- **操作对象**：`LinkRedstoneDustCoreBlock` 构造器
- **具体动作**：将默认方块状态改为同时设置 `SUPPORT_FACE=DOWN` 与 `ACTIVE=false`
- **预期结果**：默认态来源稳定，避免新状态实例继承异常 `ACTIVE`
- **关键里程碑**：是

### 步骤 2：补充状态契约测试
- **操作对象**：`LinkRedstoneDustCoreStateContractTest`
- **具体动作**：新增默认 `ACTIVE=false` 的断言测试
- **预期结果**：后续改动若破坏默认态可被自动化测试阻断
- **关键里程碑**：是

### 步骤 3：执行回归验证
- **操作对象**：Gradle 测试任务
- **具体动作**：运行 `test testIntegration testClient testSlow testApiLegacy`
- **预期结果**：确认本次修复不引入跨标签回归
- **关键里程碑**：是

## 预期结果
- 新放置或重建的核心红石粉默认 `ACTIVE` 归零，不再出现“无激活输入却常亮”的位置相关异常。
- 测试层新增默认态契约，减少同类问题复发概率。
