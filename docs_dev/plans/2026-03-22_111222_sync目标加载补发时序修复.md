# sync目标加载补发时序修复

生成时间：2026-03-22 11:12:22
文件名：2026-03-22_111222_sync目标加载补发时序修复.md

## 任务背景
当前目标区块加载时，会按 `CHUNK_LOAD -> linkAttached` 路径补发来源的同步状态。该补发事件使用的是目标区块加载时刻，而不是来源端最近一次真实同步事件的时间键。结果是在目标未加载期间，如果 later `pulse/toggle` 已进入跨区块 relay 队列，目标加载后补发的 `sync` 会被错误地视为“更新事件”，从而压制本应按时序优先覆盖它的 later 非同步事件。

同时，用户要求把“sync 目标加载补发”改成可配置开关，默认开启；本次不做兼容性设计。

## 方案详情

### 现状分析
- `LinkNodeLifecycleDispatchEvents` 在 `CHUNK_LOAD` 时直接调用 `InternalDispatchDeltaEvents.publishLinkAttached(...)`。
- `publishLinkAttached(...)` 当前仅恢复同步强度，但恢复事件的 `EventMeta` 直接沿用目标区块加载时刻。
- `LinkSyncEmitterBlockEntity` 仅持久化最近一次同步强度，`LinkSyncLeverBlockEntity` 甚至没有同步 replay 元数据。
- 目标端仲裁本身已经按时间优先生效，不应改动。

### 技术方案
- 新增配置项控制 `sync` 目标加载补发是否开启，默认 `true`。
- 为同步源实体持久化“最近一次同步 replay 快照”，至少包含：
  - `signalStrength`
  - `tick`
  - `slot`
  - `seq`
- 同步源每次真正发出 `sync` 时，先记录这份快照，再按同一时间元数据派发给目标。
- `CHUNK_LOAD` 补发时读取该快照，按来源端原始时间键恢复 `sync`，不再使用目标加载时刻。

## 原子步骤清单

### 步骤 1：新增补发开关配置
- **操作对象**：`RedstoneLinkConfig` 与 `run/config/redstonelink-server.properties`
- **具体动作**：新增 sync 目标加载补发开关，默认值为 `true`
- **预期结果**：`CHUNK_LOAD` 的 sync 恢复路径可通过配置显式关闭
- **关键里程碑**：是

### 步骤 2：同步源记录 replay 快照
- **操作对象**：同步源实体与触发路径
- **具体动作**：补充最近一次 sync 强度和时间元数据的持久化记录，并在发射器/拨杆同步派发前更新
- **预期结果**：来源端具备恢复时所需的原始 sync 时间键
- **关键里程碑**：是

### 步骤 3：恢复链路改用原始 sync 时间
- **操作对象**：`InternalDispatchDeltaEvents` 与 `LinkNodeLifecycleDispatchEvents`
- **具体动作**：目标加载补发时读取来源 replay 快照，用原始 `EventMeta` 发布恢复事件；配置关闭时跳过该补发
- **预期结果**：later `pulse/toggle` 不再被目标加载补发的旧 sync 错误压制
- **关键里程碑**：是

### 步骤 4：补测试与文档
- **操作对象**：配置解析测试、内部事件测试、使用说明
- **具体动作**：补配置默认值/显式值测试，补 sync 恢复事件应保留原始时间键的测试，并记录新配置项
- **预期结果**：新行为有回归保护，用户可查到配置用途
- **关键里程碑**：否

## 预期结果
目标区块加载时，若配置允许执行 sync 补发，则恢复事件将保留来源端最近一次真实同步的时间键；因此 later 非同步 relay 仍可按时序优先覆盖旧 sync。若配置关闭，则 `CHUNK_LOAD` 不再自动执行 sync 目标加载补发。
