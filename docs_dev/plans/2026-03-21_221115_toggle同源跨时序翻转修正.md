# toggle同源跨时序翻转修正

生成时间：2026-03-21 22:11:15
文件名：2026-03-21_221115_toggle同源跨时序翻转修正.md

## 任务背景
当前目标端在“同一来源多次 `toggle` 且时间粒度递增”的场景下，后一次 `toggle` 无法抵消前一次由同源建立的贡献，表现为目标被点亮后无法再被同源切回。

## 方案详情

### 现状分析
- `toggle` 事件在写入前会先执行“更早帧淘汰”。
- 现有实现中，同源 `toggle` 是否应抵消依赖 `findToggleContribution(sourceKey)`。
- 当 later `toggle` 先执行 `prune` 后，更早时间粒度中的同源旧贡献已经被清掉，导致系统误判为“此前没有贡献”，最终重复写入亮贡献。

### 技术方案
- 在 `toggle` 路径进入 `prune` 前，先快照“该来源是否已有 `toggle` 贡献”。
- `prune` 完成后，按快照结果决定本次 later `toggle` 是“写入新贡献”还是“抵消旧贡献”。
- 保持 `toggle` 仍然作为结构化来源贡献参与重算，不改成直接翻转目标当前状态，避免破坏失效回滚、持久化重建和并发净奇偶语义。

### 影响范围
- `src/main/java/com/makomi/block/entity/ActivatableTargetBlockEntity.java`
- `src/test/java/com/makomi/block/entity/ActivatableTargetBlockEntityInternalTest.java`

## 原子步骤清单

### 步骤 1：修正 toggle 同源 later 事件的抵消判定
- **操作对象**：`ActivatableTargetBlockEntity`
- **具体动作**：在 `toggle` 进入 `prune` 前快照同源旧贡献状态，并用该快照驱动后续写入/抵消决策
- **预期结果**：同一来源 later `toggle` 可以正确切回自己先前激活的目标
- **关键里程碑**：是

### 步骤 2：补充跨时间粒度同源 toggle 回归测试
- **操作对象**：`ActivatableTargetBlockEntityInternalTest`
- **具体动作**：新增“later toggle 抵消自己旧贡献”和“旧贡献被更晚帧清掉后再次 toggle 应重新生效”测试
- **预期结果**：覆盖本次缺陷与关键边界
- **关键里程碑**：是

## 预期结果
- later `toggle` 不再因为先 `prune` 而丢失同源旧贡献信息。
- `toggle` 仍保持结构化来源贡献语义，与 `triggerSource` 失效清理和持久化读写保持一致。
