# 重进卡0%关键路径诊断二期

生成时间：2026-03-20 18:02:26  
文件名：2026-03-20_180226_重进卡0%关键路径诊断二期.md

## 任务背景

在新建地图仅建立并激活 1 对 1 链路后，退出再进入即卡在 `Preparing spawn area: 0%`。该现象表明问题不是规模压力，而是某条关键生命周期路径在“退出/重进”边界被触发。

## 方案详情

### 现状分析

- 启动期 `CHUNK_LOAD` attach 回放已做诊断隔离后，首进可正常完成。
- 二次进入仍卡 0%，说明关键触发点更可能在“关服/卸载”与“下一次启动准备期”之间。

### 技术方案

- 在 `LinkNodeLifecycleDispatchEvents` 增加服务端生命周期状态门：
  - `SERVER_STARTED`：重置诊断计数并标记服务端进入可处理状态；
  - `SERVER_STOPPING`：标记停服中，屏蔽 `CHUNK_UNLOAD` 的 detach 回放；
  - 启动准备期（`gameTime<=0`）屏蔽全部 lifecycle 回放（load/unload）。
- 增加限流诊断日志，记录每类“被屏蔽原因”的命中。
- 保持业务语义代码不改，仅用于定位关键阻塞路径。

### 影响范围

- `src/main/java/com/makomi/data/LinkNodeLifecycleDispatchEvents.java`

## 原子步骤清单

### 步骤 1：接入服务器生命周期状态门
- **操作对象**：`LinkNodeLifecycleDispatchEvents`
- **具体动作**：注册 `SERVER_STARTED` / `SERVER_STOPPING` 事件并维护服务端状态缓存
- **预期结果**：可区分“启动准备期”“正常运行期”“停服期”
- **关键里程碑**：是

### 步骤 2：收敛区块生命周期回放入口
- **操作对象**：`publishChunkLifecycle`
- **具体动作**：依据状态门屏蔽启动准备期和停服期（仅 unload）回放，并输出诊断日志
- **预期结果**：排除 unload/prepare 关键路径对二次加载的干扰
- **关键里程碑**：是

### 步骤 3：构建与回归验证
- **操作对象**：Gradle 测试任务
- **具体动作**：执行全标签测试命令并确认构建成功
- **预期结果**：改动可编译、可回归
- **关键里程碑**：是

## 预期结果

- 若二次进入不再卡 0%，说明关键阻塞路径命中 lifecycle 回放边界。
- 若仍卡 0%，可进一步排除该路径，转向更底层加载流程（例如方块更新链或第三方混入路径）。
