# 启动卡0非阻塞回放修复

生成时间：2026-03-20 18:50:02  
文件名：2026-03-20_185002_启动卡0非阻塞回放修复.md

## 任务背景
- 最新诊断已定位：重进卡在 `Preparing spawn area: 0%` 时，服务器线程阻塞在
  `InternalDispatchDeltaEvents.resolveReplaySyncStrength -> Level.getBlockEntity -> getChunkBlocking`。
- 触发链路：启动期 `CHUNK_LOAD` 生命周期回放触发 `publishLinkAttached`，在 `SERVER_STARTED` 前执行了同步强度恢复读取。

## 方案详情

### 现状分析
- `LinkNodeLifecycleDispatchEvents` 当前仅在 `gameTime<=0` 跳过部分启动回放，仍存在启动阶段 CHUNK_LOAD 进入补发路径的窗口。
- `resolveReplaySyncStrength` 使用 `ServerLevel#getBlockEntity` / `getBlockState`，在边界时可能触发阻塞取块。

### 技术方案
1. 生命周期门控收紧：`SERVER_STARTED` 前统一跳过 `CHUNK_LOAD` 回放，避免启动期补发进入重建路径。
2. 强度恢复读取改为严格非阻塞：
   - 使用 `ServerChunkCache#getChunkNow(x,z)` 获取已就绪 chunk；
   - 仅在 chunk 已存在时通过 `LevelChunk#getBlockEntity(..., CHECK)` 和 `LevelChunk#getBlockState(...)`读取；
   - 任一条件不满足直接返回 `-1`（不补发）。

### 影响范围
- `src/main/java/com/makomi/data/LinkNodeLifecycleDispatchEvents.java`
- `src/main/java/com/makomi/data/InternalDispatchDeltaEvents.java`

### 语义对齐说明
- 命名与方向：不变更 `triggerSource/core` 语义，不改来源/目标方向规则。
- 兼容边界：仅调整启动期回放时机与读取方式，不改存档结构、键名与外部命令协议。
- API 约束：不新增 `com.makomi.api.v1.*` 依赖。
- 残余风险：启动期会减少一部分“立即补发”机会，但后续在线信号变化与队列链路仍可收敛状态。

## 原子步骤清单

### 步骤 1：启动期 CHUNK_LOAD 回放门控修正
- **操作对象**：`LinkNodeLifecycleDispatchEvents`
- **具体动作**：在 `SERVER_STARTED` 前直接跳过 `online=true` 的生命周期回放并限流记录。
- **预期结果**：启动阶段不再触发 attach 回放重建。
- **关键里程碑**：是

### 步骤 2：非阻塞强度读取
- **操作对象**：`InternalDispatchDeltaEvents.resolveReplaySyncStrength`
- **具体动作**：改为 `getChunkNow + chunk 内读取`，移除潜在阻塞调用。
- **预期结果**：不存在 `getChunkBlocking` 主线程等待。
- **关键里程碑**：是

### 步骤 3：回归验证
- **操作对象**：Gradle 测试任务
- **具体动作**：执行 `.\gradlew.bat test testIntegration testClient testSlow testApiLegacy --no-daemon`。
- **预期结果**：修复不引入回归。
- **关键里程碑**：是

## 预期结果
- 重进场景下不再在启动关键路径触发阻塞取块；
- 若来源 chunk 未就绪，则本轮补发安全跳过，待后续链路再次收敛状态。
