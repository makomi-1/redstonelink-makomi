# 目标区块加载唤醒改为按区块索引

生成时间：2026-03-22 09:54:45
文件名：2026-03-22_095445_目标区块加载唤醒改为按区块索引.md

## 任务背景

当前 `CHUNK_LOAD -> notifyTargetChunkLoaded(...)` 的唤醒实现会扫描整个 pending 快照，再筛选命中当前目标区块的不限时 pending。随着 pending 数量增加，该路径会在区块加载阶段引入线性扫描成本。用户要求实现按目标区块建立索引的优化方案。

## 方案详情

### 现状分析

- 目标区块加载唤醒当前依赖 `pendingEntriesSnapshot()` 全量扫描。
- 真正需要被唤醒的只是“已失败过、当前处于等待窗口中的不限时 pending”。
- 这些状态本来就只存在于 `DispatchState.retryStateByAttemptKey` 这类内存态中，因此优化不必侵入 `SavedData` 持久结构。

### 技术方案

- 在 `DispatchState` 中增加 `TargetChunkKey -> PendingAttemptKey 集合` 的内存索引，仅维护等待中的不限时 pending。
- 在持久 pending 失败后：
  - 计算 `nextEligibleTick`
  - 解析目标区块键
  - 将对应 `PendingAttemptKey` 写入目标区块索引
- 在以下场景移除索引：
  - pending 被接受/移除
  - 重试状态被 prune
  - 区块加载唤醒后 `nextEligibleTick` 被提前到当前时间
- `CHUNK_LOAD` 唤醒改为：
  - 先按 `TargetChunkKey` 命中候选 attempt keys
  - 再用 `queueData.pendingEntry(key)` 校验条目仍存在且版本匹配
  - 对命中的等待项直接解除等待窗口
- 不修改 `CrossChunkDispatchQueueSavedData` 的持久化结构。

### 影响范围

- `src/main/java/com/makomi/data/CrossChunkDispatchService.java`
- `src/test/java/com/makomi/data/CrossChunkDispatchServiceTest.java`

## 语义对齐说明

- 仅优化 `triggerSource -> core` 跨区块持久 pending 的区块加载唤醒路径，不改变事件语义。
- 不改变 `LinkNodeType.TRIGGER_SOURCE/CORE` 映射与方向约束。
- 不引入旧术语，内部新增索引键仅作为内存态结构使用。
- 兼容边界不变，因为不改配置键与持久化格式。

## 预期结果

`CHUNK_LOAD` 的唤醒路径将由“按全量 pending 扫描”改为“按目标区块候选集合命中”，在大量 pending 或频繁区块加载场景下显著降低唤醒成本，同时保持现有行为不变。
