# 跨区块持久重试退避阈值解耦

生成时间：2026-03-21 23:34:04
文件名：2026-03-21_233404_跨区块持久重试退避阈值解耦.md

## 任务背景
当前跨区块持久队列中的不限时 `SYNC_SIGNAL` 会长期重试，这是预期行为；但现有实现把“非持久事件丢弃阈值”和“持久事件进入退避重试的起点”复用了同一个 `dropThreshold`，同时还会对持久事件触发通用 `warn/error` 重试日志，导致正常长期等待目标加载的场景也可能被误报为错误。

## 方案详情

### 现状分析
- 持久 `SYNC_SIGNAL` 会生成不限时 pending（`expireGameTick=Long.MAX_VALUE`），长期重试属于设计语义。
- 当前 `shouldBackoffRetry(...)` 与 `recordRetryFailureAndShouldDrop(...)` 都以 `crosschunk.retry.dropThreshold` 作为持久 pending 的退避起点。
- 当前 `warn/error` 阈值对不限时 pending 同样生效，不利于区分“异常失败”和“正常长期等待”。

### 技术方案
- 新增独立配置 `crosschunk.retry.persistentBackoffStartThreshold`，专门控制持久 pending 从第几次失败开始进入退避；`0=关闭退避限制`。
- 保留现有 `crosschunk.retry.persistentBackoffTicks`，继续控制进入退避后的重试间隔。
- `crosschunk.retry.dropThreshold` 保持只服务“非持久事件达到上限后丢弃”。
- 不限时 pending 默认不再触发通用 `warn/error` 阈值日志，仅在首次进入退避时输出一次提示日志。

## 预期结果
- 持久 `SYNC_SIGNAL` 继续长期重试，但进入退避的起点由独立配置控制。
- 非持久事件的丢弃阈值与持久事件的退避阈值不再耦合。
- 正常的长期等待目标加载不再默认输出 `warn/error` 重试误报，只在首次进入退避时给出一次提示。
