### java版游戏出生点附近的区块是常驻加载的

### 游戏更新机制

不能简单理解成“游戏每 tick 末统一结算一次再对外生效”。

更准确地说，Minecraft 服务端是“按 tick 推进”，但 tick 内很多逻辑是立即执行的：
- 方块状态写回是立即的
- 邻居通知是立即链式扩散的
- 方块实体逻辑、命令、事件投影也都是当前 tick 内同步跑完

所以同一 tick 内如果目标端先后收到多条 delta，哪怕最终态一致，中间态理论上也可能被别的方块/逻辑观察到。  
这也是我前面说“tick 内波动不一定完全不可见”的原因。

你如果只看玩家肉眼体验，通常不明显，因为：
- 客户端渲染帧和网络同步不一定把所有中间态都展示出来
- 很多中间变化发生在同一个服务端 tick 内

但如果看红石/邻居更新语义，就不能当它不存在。当前实现里尤其是 `toggle/pulse` 路径，并没有像 `sync` 那样做同时间粒度扇出去重，所以同 tick 多次不同结果，外部是有机会感知到的。

所以结论应改成一句话：  
“游戏是按 tick 驱动，但不是按 tick 末一次性提交；同 tick 中间态在服务端逻辑层可能真实生效。”

如果你要的语义是“同一 `timeKey` 只允许最终结果对外生效一次”，那就得做“同帧缓冲后统一提交”，不是靠现在这种逐条 delta 增量处理自然得到的。


### 启动期信道补发必须避免阻塞取块
- 解决要点：
1. 问题：世界重进卡在 `Preparing spawn area: 0%`，根因是启动期 `CHUNK_LOAD` 回放触发 `sync` 恢复读取，调用 `getBlockEntity/getBlockState` 间接进入 `getChunkBlocking`，把服务器线程阻塞在 `prepareLevels`。
2. 处理：`SERVER_STARTED` 前统一跳过 `CHUNK_LOAD` 回放；`resolveReplaySyncStrength` 改为非阻塞读取（`getChunkNow + EntityCreationType.CHECK`），未就绪即返回 `-1` 跳过补发。
3. 可复用结论：启动关键路径禁止任何可能触发“同步等待 chunk 就绪”的读取；恢复补发应采用“可得即补、不可得即跳过并等待后续收敛”的非阻塞策略。

### 断点阻塞会制造“更新风暴”假象
- 解决要点：
1. 热点路径（如 `neighborChanged`、`scheduleTick`、`setBlock`）一旦命中可挂起断点，会直接阻塞主线程，外观上会表现为“更新堆积”和“性能骤降”。
2. 先确认是否有断点处于 `Suspend` 状态；若关闭断点后现象消失，应优先判定为调试干扰而非逻辑回归。
3. 高频路径优先使用日志断点或无副作用条件断点，避免在性能排查阶段使用停机断点制造伪结论。

### 断点条件可使用方法返回值，但应限制为无副作用表达式
- 解决要点：
1. 条件断点不仅可用局部变量，也可用可求值表达式，例如 `getSerial()`、`worldPosition.getX()`、`state.getValue(...)`。
2. 条件表达式应只调用无副作用方法（getter/纯计算）；避免调用会改状态的方法，防止调试过程改变运行时行为。
3. 方法求值有额外开销，复杂条件在高频路径会拖慢 tick；实战上建议先用“序号+坐标”做粗筛，再加状态条件细化。

### 退役副作用收口优先放在协调层入口，而非 `retireNode()` 本体
- 解决要点：
1. `retireNode()` 应保持为底层数据原语，仅处理退役主事实（标记退役、移除在线节点、清理链接），避免与白名单策略强耦合。
2. 统一由协调层入口（如 `retireAndSyncWhitelist(level, type, serial)`）编排“先退役、后强同步清理白名单”，可保证所有调用路径副作用一致。
3. 协调层更便于后续扩展审计、告警、补偿等横切逻辑，降低在多处调用点遗漏副作用的风险。

### 工具类使用 `private` 空构造器是防实例化约定，不是漏实现
- 解决要点：
1. 仅提供 `static` 能力的类（如协调器/工具类）不需要对象实例，应通过 `private` 构造器阻止 `new`。
2. 空构造器用于清晰表达“此类只作为静态入口”，提升可读性与使用约束一致性。
3. 该写法可避免误用实例状态，减少不必要对象创建与维护成本。

### 命名迁移采用“新入口先落地 + 旧壳短期并存 + 去壳收口”过渡法
- 解决要点：
1. 第一阶段先新增语义主入口（如 `TriggerSourcePairingScreen`、`OpenTriggerSourcePairingPayload`），保证新实现有统一落点。
2. 第二阶段保留旧命名壳并委托新入口，用定向测试验证“仅命名变化、行为不变”。
3. 第三阶段在确认仅内部调用后执行去壳清理，降低长期维护成本与 deprecated 告警噪音。

### 跨重启强制加载清理是跨区块策略的前置保障
- 解决要点：
1. 问题现象：跨区块提示偶发失效、白名单看似失效，且“未主动释放”后可跨重启复现。
2. 根因定位：仅依赖运行期过期释放不足，停服阶段若不兜底释放 `forced chunk`，会把已强制加载区块残留到下次启动环境。
3. 可复用结论：凡是有“运行时申请资源 + 延迟释放”模式（如强制加载、临时票据、缓存接管），都必须补齐停服/退出路径的统一清理回调，避免状态跨重启泄漏。

### 终端启动 + IDE 附加调试时的 Lithium 判定
- 解决要点：
1. 仅执行 `./gradlew -PwithLithium=true --no-daemon runClient --debug-jvm` 并不代表当前日志一定是该次进程；若 `5005` 端口被旧 Java 进程占用，新进程会失败，`latest.log` 仍可能停留在上一次“无锂”启动结果。
2. VSCode 必须使用 `Attach Gradle Debug (5005)` 附加到同一进程，避免误触发 `Minecraft Client` 直启配置导致走无锂链路。
3. 每次进游戏后立即以日志二次确认：`Loading 56 mods` 且 `lithiumLoaded=true`；若未同时满足，优先排查“进程是否同一条启动链”和“端口是否被占用”。

### TreeSet 在序号展示链路中的性质与应用
- 解决要点：
1. 选择 `TreeSet` 的核心原因是“去重 + 升序”一步完成，便于后续做连续区间压缩（`A:B`）。
2. 底层连接关系语义是集合（无序），展示层若不排序会出现同数据不同次显示顺序漂移；`TreeSet` 保证展示稳定。
3. 适用场景：展示需要稳定顺序且要做区间合并；不适用场景：需要保留原始输入顺序或允许重复的业务语义。

### 遥控器注册参数差异源于物品类型
- 解决要点：
1. 问题：遥控器注册方法与其他物品看似不同，容易被误判为流程不一致。
2. 解决要点：LinkerItem 是纯物品，需注入 `ActivationMode`；PairableBlockItem 依赖方块与 `LinkNodeType`，构造参数不同导致注册形式不同。
3. 可复用结论：注册差异优先从“物品类型 + 构造参数”解释，避免为了形式统一而破坏类型语义。

### 通用层避免直接依赖客户端截断逻辑
- 解决要点：
1. 问题：物品栏 tooltip 需要截断，但 GUI 逻辑依赖客户端渲染类，通用层直接引用会导致服务端加载报错。
2. 在 `com.makomi.item` 新增轻量截断工具，按字符数截断并追加 `…(+N)`。
3. 可复用结论：通用层避免引用客户端类；长文本统一走工具类集中处理。

### 经验模块索引收敛
- 解决要点：
1. 将后续迭代中高复用经验整理为固定模块，便于跨任务复用。
2. 经验模块与主文档建立一对一映射，减少重复维护。
3. 后续新增经验优先补充到 `docs/experience` 与 `docs/runbooks`，本文件保留索引性质。

### 长列表显示与可读性控制
- 解决要点：
1. 对“当前连接”类长列表采用“宽度截断 + `…(+N)`”策略，避免主界面拥挤。
2. tooltip 也需限制最大条目数并展示剩余数量，保证可读性与信息提示平衡。
3. 将截断与提示逻辑集中到通用渲染路径，减少多处重复实现。
