# 版本号设计参考（按 05 路线）

> 更新日期：2026-03-09  
> 适用范围：`RedstoneLink`（Minecraft 1.21.1+，Fabric 生态）

## 1. 目标

- 保持版本号可读：看版本即可判断稳定度与风险。
- 保持升级可追溯：每次升版都有明确依据与测试证据。
- 保持发布可执行：与 `05` 里程碑、兼容矩阵、风险清单对齐。

## 2. 版本格式

推荐格式：

`MAJOR.MINOR.PATCH[-PHASE.SEQ]`

示例：

- `1.0.0-alpha.1`
- `1.0.0-beta.2`
- `1.0.0-rc.1`
- `1.0.0`

字段说明：

- `MAJOR`：不兼容变更（API/存档/核心行为）。
- `MINOR`：向后兼容的新能力与扩展。
- `PATCH`：向后兼容的修复与稳定性改进。
- `PHASE`：`alpha` / `beta` / `rc`。
- `SEQ`：同阶段迭代序号。

## 3. 与 05 的门禁映射

### alpha（开发验证阶段）

- 目标：功能可跑通，关键链路可诊断。
- 准入参考：
  - 05 的 M1/M2 达成；
  - E0/E1 回归通过；
  - 无阻断级启动或注入异常。

### beta（生态兼容扩展阶段）

- 目标：跨模组组合稳定，行为一致性增强。
- 准入参考：
  - 05 的 M3 进入稳定推进；
  - E0~E7 至少完整通过 1 轮；
  - 风险清单无新增高优先级阻断项。

### rc（冻结前候选阶段）

- 目标：发布前收敛风险与证据。
- 准入参考：
  - 05 的 M4 启动并收敛脚本化回归；
  - 扩展矩阵 27 组通过；
  - 关键风险项完成复核。

### 正式版（release）

- 目标：可对外稳定发布。
- 准入参考：
  - 版本冻结前回归完成；
  - 证据归档完整（日志/矩阵结果/关键行为截图或录像）。

## 4. 升版判定规则

### PATCH 升版（`x.y.z -> x.y.(z+1)`）

满足任一：

- bug 修复（逻辑、兼容、构建、资源）且不破坏兼容性；
- 注入匹配、优先级、诊断链路等稳定性修复；
- 配方或资源纠错且语义保持兼容。

### MINOR 升版（`x.y.z -> x.(y+1).0`）

满足任一：

- 新增可见功能或可配置能力；
- 新增 API 能力且向后兼容；
- 兼容矩阵扩展到新类别并成为发布承诺的一部分。

### MAJOR 升版（`x.y.z -> (x+1).0.0`）

满足任一：

- API 不兼容；
- 存档结构不兼容或迁移不可逆；
- 默认行为发生破坏性改变，旧版本配置无法平滑过渡。

## 5. 变更后自动审查流程（执行规范）

每次完成修改后，按以下顺序执行一次版本号审查：

1. 归类变更范围：功能新增 / 修复 / 兼容 / 资源 / 构建。
2. 判定兼容影响：是否涉及 API、存档、默认行为破坏。
3. 对照门禁：当前结果落在 alpha/beta/rc/release 的哪个阶段。
4. 形成建议：`不升版` / `PATCH` / `MINOR` / `MAJOR`。
5. 询问用户：若存在升版建议，必须先请求用户确认再改 `mod_version`。

## 6. 用户确认时的最小依据模板

```text
建议升版：PATCH（示例）
依据：
1) 变更类型：兼容修复 + 资源纠错（无 API/存档破坏）
2) 影响范围：仅影响 XXX 文件与 XXX 行为
3) 回归结果：E0/E1 通过（或当前已完成的矩阵项）
4) 风险评估：无新增阻断风险
请确认是否更新版本号（当前 x.y.z -> 建议 x.y.(z+1)）？
```

## 7. 本项目落点

- 当前版本源：`gradle.properties` 中 `mod_version`。
- 构建读取：`build.gradle` 的 `version = project.mod_version`。
- 版本更新动作：默认手工改值，但必须先完成“自动审查 + 用户确认”。
